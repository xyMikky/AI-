# 智能体逻辑链路诊断（Logic Chain Diagnostic）

## 概述

本 Skill 为「AI设计师助手」提供**系统级自检能力**——在不执行具体设计任务的情况下，全面扫描智能体的所有组件（模块、规则、Skill、参考库、品牌规范、场景知识库）并验证它们之间的调用链路是否畅通。

**V8.0 角色继承**：原 M18_逻辑链路诊断 业务模块已合并入本 Skill，本 Skill 同时承担"系统维护工具"和"业务模块"双重身份，通过同一份 `DiagnosticConfig` + 12 维检查输出统一报告。

**核心理念**：一个复杂智能体系统中，任何一个文件缺失、路径错误、注册不一致都可能导致某个功能在运行时无法被正常调用。本诊断助手就是这条保险线——在问题发生前发现隐患。

## 使用场景

| 场景 | 用户表述示例 |
|------|-------------|
| 系统健康检查 | "检查一下系统""诊断链路""看看有没有问题" |
| 新增模块后验证 | "刚加了 M18，检查下是否正确注册" |
| 学习引擎后自检 | "刚跑完学习，检查参考库索引是否一致" |
| 品牌初始化后验证 | "初始化了新品牌，检查目录是否完整" |
| 规则变更后验证 | "修改了规则文件，检查交叉引用是否正常" |
| 定期巡检 | "跑一遍全面诊断" |
| 局部检查 | "只检查品牌规范""只检查参考库""只检查 Skill" |

## 12 维诊断维度

| 维度 | 检查内容 | 严重度 |
|------|---------|--------|
| **D1** 核心文件 | 主控中心.txt、AS-AI角色架构师.txt 是否存在 | 致命 |
| **D2** 模块注册 | M1-M17 模块文件是否存在且与注册表一致 | 致命 |
| **D3** 规则文件 | .codex/rules/ 下所有 .mdc 文件完整性 | 高 |
| **D4** Skill 完整性 | 每个 Skill 的 SKILL.md + 脚本文件 | 高 |
| **D5** 参考库 | 域目录+域索引+参考档案+通识原则 | 中 |
| **D6** 品牌规范 | 品牌目录结构、模板、已注册品牌的文件 | 中 |
| **D7** 场景知识库 | K1-K4 目录和知识文件 | 中 |
| **D8** 人物/场景库 | 子目录结构和索引文件 | 中 |
| **D9** 主流程链路 | design-must-use-main-control 主流程每个阶段门的依赖文件 | 致命 |
| **D10** 交叉引用 | 规则间相互引用的文件是否都存在 | 高 |
| **D11** 触发词路由 | 触发关键词能否正确路由到目标模块 | 高 |
| **D12** 原始素材归档 | 各域的原始素材目录是否存在 | 低 |

## 工作流程

### 步骤 1：运行诊断脚本

```bash
python .codex/skills/logic-chain-diagnostic/scripts/diagnostic_agent.py
```

**可选参数**：

| 参数 | 作用 | 示例 |
|------|------|------|
| `--root` | 指定项目根路径（默认自动检测） | `--root "f:\AI设计师助手"` |
| `--json` | 输出 JSON 格式（用于程序化处理） | `--json` |
| `--dimension` | 仅检查指定维度 | `--dimension d6`（仅检查品牌规范） |

### 步骤 2：阅读诊断报告

脚本输出一份结构化诊断报告，包含：
- **系统健康度百分比**（🟢 ≥95% / 🟡 80-95% / 🟠 60-80% / 🔴 <60%）
- **按维度分组的检查结果**（只展开失败和警告项）
- **关键问题清单**（必须修复的 ❌ 项）
- **改进建议清单**（可选修复的 ⚠️ 项）

### 步骤 3：根据问题修复

对报告中的每个问题项：
- **❌ 致命/高**：立即修复（创建缺失文件、更新注册表、修正路径）
- **⚠️ 警告**：按优先级处理（补充索引、注册新增组件等）
- **✅ 通过**：无需操作

### 步骤 4：修复后复查

修复完成后再次运行脚本，确认所有问题已解决。

## 注意事项

- 诊断脚本是**只读**的，不会修改任何文件
- 部分 ⚠️ 警告是正常的（如某些域设计上不需要域索引）
- 新增模块/规则/Skill 后应同步更新诊断脚本中的 `DiagnosticConfig` 配置
- 建议每次系统变更（新增模块/修改规则/初始化品牌）后运行一次诊断
