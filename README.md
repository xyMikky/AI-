# AI 工作助手

公司内部 AI 智能体与 Skill 工程文件仓库。这个项目把设计、电商、Amazon 竞品分析、品牌规范、图片生成、视频脚本、工作流沉淀等能力整理成一套可复用的本地 AI 工作助手。

当前仓库主要面向 **Cursor / AI IDE 工作流** 使用，核心配置位于 `.cursor/skills` 和 `.cursor/rules`。

## 给普通同事：怎么用

如果你不熟悉 GitHub，不需要学习 GitHub。请使用维护者提供的压缩包或公司共享盘版本。

### 第一次使用

1. 从公司共享盘、群文件或维护者发来的压缩包下载最新版。
2. 解压到电脑本地，例如：

```text
D:\AI工作助手
```

3. 用 Cursor 打开这个文件夹。
4. 等 Cursor 读取项目配置后，直接对 AI 说你的需求，例如：

```text
帮我分析这个 Amazon 竞品链接
帮我做一张品牌海报
帮我优化这个产品详情页结构
帮我根据这些图片学习品牌风格
```

### 更新版本

普通同事不需要操作 GitHub。推荐使用下面任一方式更新：

- 重新下载维护者发布的最新版压缩包。
- 从公司共享盘复制最新版文件夹。
- 如果维护者提供了“一键更新”脚本，双击运行即可。

更新后建议重启 Cursor，让新的 Skill 和规则生效。

## 环境准备

普通同事通常只需要：

- Windows 10 或更高版本
- Cursor
- 可以访问公司内部共享盘或维护者发布的压缩包

部分高级能力可能需要：

- Python 3.10+
- API Key，例如 RunningHub、SiliconFlow 或内部数据服务
- 公司网络或指定数据权限

如果你只是让 AI 写方案、分析文本、整理思路，通常不需要额外配置。

## API Key 与敏感信息

不要把真实 API Key 提交到 GitHub。

本项目使用：

```text
config/.env.example
```

作为配置模板。真实密钥应放在：

```text
config/.env
```

`config/.env` 已被 `.gitignore` 忽略，不应进入仓库。

## 给维护者：如何发布给不会 GitHub 的同事

推荐流程：

1. 在本地维护项目内容。
2. 提交并推送到 GitHub，作为内部源仓库。
3. 从 GitHub 下载 ZIP，或从本地打包一份干净版本。
4. 发布到公司共享盘、飞书 / 企业微信文件、OneDrive 或 NAS。
5. 通知同事下载最新版并重启 Cursor。

建议发布包命名：

```text

```

例如：

```text

```

## 给维护者：更新 Skill 的建议流程

新增或修改 Skill 时，建议按下面顺序检查：

1. 确认 Skill 位于 `.cursor/skills/[skill-name]/`。
2. 确认存在 `SKILL.md`。
3. `SKILL.md` 的 description 要写清楚触发场景。
4. 如有脚本，放入对应 Skill 的 `scripts/`。
5. 如有参考资料，放入对应 Skill 的 `references/`。
6. 如有图片、模板、示例，放入 `assets/`。
7. 如果需要全局触发，更新 `.cursor/rules/` 中对应规则。
8. 更新本 README 或内部发布说明。

## 常见问题

### 同事不会 GitHub 怎么办？

不要让普通同事用 GitHub。维护者用 GitHub 管理版本，普通同事使用 ZIP 包或公司共享盘版本即可。

### 为什么更新后 AI 没有识别新 Skill？

通常是 Cursor 还没有重新加载项目。请关闭并重新打开 Cursor，或重新打开当前项目文件夹。

### 可以直接把整个仓库发给别人吗？

可以，但发布前请确认没有包含：

- `config/.env`
- 个人账号信息
- 未授权的客户资料
- 不应公开的图片素材
- 大体积临时输出文件

### 这个项目能不能给 Codex 用？

当前结构主要是 Cursor 项目结构。如果要作为 Codex Skill 分发，建议维护者额外制作安装脚本，把需要复用的 Skill 同步到用户本机的 Codex skills 目录。不要让普通同事手动处理这些路径。

## 维护原则

- GitHub 用于内部版本管理。
- 普通同事只接触压缩包、共享盘或一键安装入口。
- README 面向第一次接触项目的人，尽量少写内部黑话。
- API Key、客户资料、生成结果和大文件默认不入库。
- 新增能力要优先沉淀成 Skill，而不是散落在聊天记录里。
